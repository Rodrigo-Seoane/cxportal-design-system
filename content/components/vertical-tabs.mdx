## Overview

Vertical Tabs provide section-level navigation within a bounded surface such as a settings panel, a detail drawer, or a sidebar. Unlike horizontal tabs, they scale gracefully to long label text and accommodate a higher number of items without overflow. Each tab is a full-width button; only one tab is active at a time.

The component ships as two primitives:

- **`VerticalTab`** — a single interactive tab item with icon, label, and an optional right-side slot for count chips or status badges.
- **`VerticalTabGroup`** — a container that stacks tabs vertically with consistent spacing and a white background card.

---

## When to use

- Navigating between distinct sections within a panel, drawer, or settings page
- When the number of sections is between 2 and 7 — fewer than 2 tabs is not meaningful; more than 7 becomes hard to scan
- When labels are long or variable-length and would not fit comfortably in a horizontal tab bar
- When tabs need to carry secondary information such as record counts or status badges

**Do not use Vertical Tabs when:**
- Switching between content views at the page level — use the top navigation or horizontal tabs
- The sections are deeply nested — prefer a breadcrumb or stepped layout instead
- Only a single item exists — a heading with no alternatives is not a tab

---

## Anatomy

- **Group container** — white background (`#ffffff`), 8 px padding, 4 px gap between items, 8 px border radius. Renders as `role="tablist"` with `aria-orientation="vertical"`.
- **Tab button** — 14 px label, 16 px horizontal padding, 12 px vertical padding, 8 px border radius. Full-width within its container.
- **Icon** — optional 16 × 16 px leading element. Matches the current text colour for each state.
- **Right slot** — optional trailing region for count chips or status badges. Sits flush to the right edge.
- **Active indicator** — solid blue fill (`#4285f4`) replaces the transparent background; label colour inverts to `#eff1f3` and weight increases to SemiBold (600).

## Do's & Don'ts

<DosDonts>
  <Do description="Keep tab labels short and parallel — one to three words each. Users scan the list quickly; long or inconsistent labels slow navigation." />
  <Dont description="Mix navigational tabs and action buttons in the same VerticalTabGroup. The group is a navigation landmark; adding unrelated actions creates confusion." />
</DosDonts>

<DosDonts>
  <Do description="Use the right-side slot for read-only context (record counts, 'New' badges). Clicking the tab always navigates — the slot is not a separate click target." />
  <Dont description="Use Vertical Tabs for progressive disclosure or accordions. Each tab should reveal a full section, not toggle a panel inline." />
</DosDonts>

<DosDonts>
  <Do description="Show the icon consistently across all tabs in a group, or not at all. A mix of icon and no-icon tabs breaks the visual rhythm." />
  <Dont description="Use more than 7 tabs in a single group. Beyond that, consider a nested navigation pattern or a searchable list." />
</DosDonts>

## Code

### Basic group

```tsx
import { VerticalTab, VerticalTabGroup } from '@/components/ui/vertical-tabs'

const [active, setActive] = useState('production')

<VerticalTabGroup>
  <VerticalTab
    label="Global Permissions"
    active={active === 'global'}
    onClick={() => setActive('global')}
  />
  <VerticalTab
    label="Instances"
    active={active === 'instances'}
    onClick={() => setActive('instances')}
  />
  <VerticalTab
    label="Production"
    active={active === 'production'}
    onClick={() => setActive('production')}
  />
</VerticalTabGroup>
```

### With icons

```tsx
import { ShieldIcon } from '@phosphor-icons/react'

<VerticalTabGroup>
  <VerticalTab
    label="Global Permissions"
    icon={<ShieldIcon size={16} />}
    active
    onClick={() => {}}
  />
  <VerticalTab
    label="Instances"
    icon={<ShieldIcon size={16} />}
    onClick={() => {}}
  />
</VerticalTabGroup>
```

### With count badge in right slot

```tsx
import { Chip } from '@/components/ui/chip'

<VerticalTabGroup>
  <VerticalTab
    label="Categories"
    active
    rightSlot={<Chip label="14" type="count" shade={1} />}
    onClick={() => {}}
  />
  <VerticalTab
    label="Tags"
    rightSlot={<Chip label="6" type="count" shade={1} />}
    onClick={() => {}}
  />
</VerticalTabGroup>
```

### Disabled tab

```tsx
<VerticalTabGroup>
  <VerticalTab label="Production" active onClick={() => {}} />
  <VerticalTab label="Development" onClick={() => {}} />
  <VerticalTab label="Q&A" disabled onClick={() => {}} />
</VerticalTabGroup>
```

### Controlled with state

```tsx
const SECTIONS = ['Global Permissions', 'Instances', 'Production', 'Development', 'Q&A'] as const
type Section = typeof SECTIONS[number]

const [active, setActive] = useState<Section>('Production')

<VerticalTabGroup style={{ width: 240 }}>
  {SECTIONS.map((label) => (
    <VerticalTab
      key={label}
      label={label}
      active={active === label}
      onClick={() => setActive(label)}
    />
  ))}
</VerticalTabGroup>
```

## Accessibility

- `VerticalTabGroup` renders a `<div role="tablist" aria-orientation="vertical">` so assistive technologies understand this is a vertical tab navigation.
- Each `VerticalTab` renders a `<button role="tab">` with `aria-selected="true"` on the active item.
- Disabled tabs receive both the native `disabled` attribute and `aria-disabled="true"` so their state is conveyed to all assistive technologies.
- The active tab should be paired with a corresponding `<div role="tabpanel">` in the main content area. Connect them with matching `id` and `aria-labelledby` attributes so screen readers can announce which panel is visible.
- Tab panels that are not selected should be hidden with `hidden` or `display: none` — do not render all panels simultaneously.
- Keyboard navigation follows the ARIA Tabs pattern: Tab moves focus into and out of the tab list; Arrow Up / Arrow Down move between tabs; Enter or Space activates the focused tab.
