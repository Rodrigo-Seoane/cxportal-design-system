## Overview

Toast notifications are short, non-intrusive messages that appear temporarily to confirm an action, communicate a status change, or surface a warning. They stack at a fixed edge of the viewport, auto-dismiss after a set duration, and never block the underlying UI.

CxPortal toasts support six types — default, success, error, warning, info, and loading — with the option to include a secondary description line and a single action button for reversible operations.

## When to use

**Use toasts for:**

- Confirming that a user action was completed (file saved, record deleted, form submitted)
- Non-critical errors that don't prevent the user from continuing (network retry, sync failure)
- Warnings about side effects that don't require an immediate decision
- Surfacing the progress or outcome of an async operation (`toast.promise`)
- Offering a brief undo window after a destructive action

**Do not use toasts for:**

- Critical errors that block the workflow — use a Modal or inline error instead
- Decisions that require the user to choose between options — use a Dialog
- Form field validation — use inline error messages beneath each field
- Permanent or persistent status information — use a Badge or MessageBox

## Behaviour

### Position

The default position is **top-right**. This keeps notifications away from primary content areas (forms, tables) while remaining in the user's natural reading flow. Other positions are supported for layout requirements.

Override the duration with the `duration` option (in milliseconds). Pass `duration: 0` to prevent auto-dismiss entirely.

### Animation

Toasts slide in from the edge of the viewport (16 px horizontal offset) with 200 ms ease transitions on `opacity` and `transform`. On dismiss the animation reverses. The 300 ms prune delay ensures the exit animation completes before the element is removed from the DOM.

### Stacking

The `Toaster` shows the most recent `maxToasts` toasts (default: 3). Toasts beyond this limit are queued in the store and displayed as older ones dismiss. The newest toast always appears visually closest to the viewport edge.

## Setup

Add a single `<Toaster />` to your root layout. It does not need to be near the component that triggers the toast — the store is module-level.

```tsx
// app/layout.tsx
import { Toaster } from '@/components/ui/toast'

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        {children}
        <Toaster position="top-right" />
      </body>
    </html>
  )
}
```

## API reference

### Imperative API

Call `toast` or any of its methods from anywhere — server actions, event handlers, utility functions.

```tsx
import { toast } from '@/components/ui/toast'

// Default
toast('Changes saved.')

// Typed
toast.success('File uploaded successfully.')
toast.error('Upload failed. Please try again.')
toast.warning('Session expires in 5 minutes.')
toast.info('A new version is available.')

// With description
toast.success('Event created', {
  description: 'Sunday, December 03, 2023 at 9:00 AM',
})

// With action button
toast.error('Message deleted', {
  action: { label: 'Undo', onClick: () => restoreMessage() },
})

// Custom duration (ms). Pass 0 to prevent auto-dismiss.
toast.info('Syncing…', { duration: 0 })

// Dismiss programmatically
const id = toast.loading('Uploading…')
toast.dismiss(id)

// Dismiss all
toast.dismiss()
```

### toast.promise

Automatically transitions a loading toast to success or error based on a promise's outcome.

```tsx
toast.promise(uploadFile(file), {
  loading: 'Uploading file…',
  success: (data) => `${data.filename} uploaded successfully.`,
  error:   (err)  => `Upload failed: ${err.message}`,
})
```

The `loading`, `success`, and `error` values accept a plain string or a function that receives the resolved value / rejection reason.
