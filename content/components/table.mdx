## Overview

The Table component displays structured, multi-column data in a scannable grid. It supports two density modes — **Wide** (comfortable, 64 px rows) and **Compact** (dense, 40 px rows) — along with sortable headers, row selection via checkboxes, zebra striping, and a rich set of cell content types (text, links, tags, action buttons, switches, and more).

## When to use

- When presenting **6 or more rows** of related, structured data
- When users need to **compare values across rows** (tabular data)
- When **bulk operations** (select, delete, export) on multiple items are required
- When users need to **sort or filter** a dataset to find specific entries

**Do not use** a table for non-tabular content, simple key-value pairs (prefer a description list), or when fewer than 3 columns would be displayed.

## Principles

### Scanability first
Organise columns by importance, placing key identifiers (names, IDs) on the left where users begin scanning. Use consistent alignment and adequate spacing to create clear visual lanes.

### Progressive disclosure
Show essential information in the default view and provide mechanisms (expandable rows, detail panels, modals) for accessing additional data without cluttering the table.

### Actionable context
Place row actions where users expect them — typically the rightmost column — and provide clear affordances for interactive elements. Batch actions should surface when multiple rows are selected.

### Visual hierarchy
Use typography weight, colour, and spacing to differentiate between headers, data cells, and interactive elements. Status indicators and tags should be immediately recognisable without relying on colour alone.

## Do's & Don'ts

<DosDonts>
  <Do description="Keep column count manageable — 5–8 columns is ideal on desktop. If more data is needed, use progressive disclosure (expandable rows, side panels) rather than adding more columns." />
  <Dont description="Overcrowd cells with too much information. Each cell should carry one primary value. Secondary context belongs in a tooltip, sub-row, or detail panel." />
</DosDonts>

<DosDonts>
  <Do description="Align data types consistently — numbers flush right, text flush left, status indicators centred. This creates natural visual lanes that users can scan vertically." />
  <Dont description="Mix alignments within a column or centre-align long text values. Ragged left edges on text make vertical scanning significantly harder." />
</DosDonts>

<DosDonts>
  <Do description="Use the Compact density when vertical space is limited or when users are working with dense operational data (admin panels, dashboards). Compact rows (40 px) pack more rows above the fold." />
  <Dont description="Switch between Wide and Compact on the same page or within a single session without a user preference. Inconsistent density is disorienting." />
</DosDonts>

<DosDonts>
  <Do description="Show the indeterminate state (dash) on the header checkbox when some — but not all — rows are selected. This accurately communicates partial selection before the user commits to select-all." />
  <Dont description="Skip the indeterminate state by jumping directly from unchecked to fully checked. Users lose track of whether their partial selection has been replaced." />
</DosDonts>

<DosDonts>
  <Do description="Enable zebra striping (`striped` prop on alternating TableRows) for wide tables with many columns. Alternating backgrounds (#f8f8f8) guide the eye horizontally across dense rows." />
  <Dont description="Use striping on narrow tables with 3 or fewer columns — it adds visual noise without the scanability benefit." />
</DosDonts>

<DosDonts>
  <Do description="Limit visible row actions to 3 icon buttons maximum (Edit, View, Delete). Overflow additional actions into a DotsThree menu to keep the actions column compact." />
  <Dont description="Place destructive actions (Delete) as the primary/first visible button. Always position Edit or View first; Delete should require confirmation and be the last action." />
</DosDonts>

### Sortable columns

```tsx
const [sort, setSort] = useState<{ col: string; dir: 'asc' | 'desc' | 'none' }>({
  col: '', dir: 'none',
})

function handleSort(col: string) {
  setSort(prev =>
    prev.col !== col
      ? { col, dir: 'asc' }
      : prev.dir === 'asc'
      ? { col, dir: 'desc' }
      : { col: '', dir: 'none' },
  )
}

<TableHead
  sortDirection={sort.col === 'name' ? sort.dir : 'none'}
  onSort={() => handleSort('name')}
>
  Name
</TableHead>
```

### Selectable rows with batch action

```tsx
import {
  Table, TableHeader, TableBody, TableRow,
  TableHead, TableCell, TableCheckboxHead, TableCheckboxCell,
} from '@/components/ui/table'

const [selected, setSelected] = useState<Set<number>>(new Set())

const allSelected = selected.size === rows.length
const someSelected = selected.size > 0 && !allSelected

function toggleAll(checked: boolean) {
  setSelected(checked ? new Set(rows.map(r => r.id)) : new Set())
}

<Table size="wide">
  <TableHeader>
    <tr>
      <TableCheckboxHead
        checked={allSelected}
        indeterminate={someSelected}
        onChange={toggleAll}
      />
      <TableHead>Name</TableHead>
      <TableHead>Status</TableHead>
    </tr>
  </TableHeader>
  <TableBody>
    {rows.map(row => (
      <TableRow key={row.id} selected={selected.has(row.id)}>
        <TableCheckboxCell
          checked={selected.has(row.id)}
          onChange={v => {
            const next = new Set(selected)
            v ? next.add(row.id) : next.delete(row.id)
            setSelected(next)
          }}
        />
        <TableCell>{row.name}</TableCell>
        <TableCell>{row.status}</TableCell>
      </TableRow>
    ))}
  </TableBody>
</Table>
```

## Accessibility

- Use semantic `<table>`, `<thead>`, `<tbody>`, `<th>`, `<td>` elements — the component handles this automatically.
- `<th scope="col">` is set on every `TableHead` cell so screen readers can associate headers with their columns.
- Sortable columns expose `aria-sort` (`"ascending"`, `"descending"`, or `"none"`) so sort state is announced when focus lands on the header.
- `TableRow` sets `aria-selected` on selected rows so selection state is announced.
- `TableCheckboxHead` carries `aria-label="Select all rows"`; pass a meaningful `ariaLabel` to `TableCheckboxCell` for each row (e.g. `"Select Dianne Russell"`).
- The indeterminate state is set via `HTMLInputElement.indeterminate` (not a HTML attribute) — the component handles this correctly with a `useRef` + `useEffect` pattern.
- Ensure a minimum 44 × 44 px touch target for all interactive elements in action columns.
- Maintain a 4.5 : 1 contrast ratio for all cell text; the design tokens meet this threshold.
- Visible focus rings are inherited from the global focus styles — do not suppress `outline` on table rows or cells.
- Announce selection count changes to screen readers by pairing the table with a live region (e.g. `aria-live="polite"`) in the toolbar that reads "3 of 12 rows selected".
